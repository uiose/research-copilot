## 二、过程记录
### 我做了什么
- 对project_definition.md进行了完善，确定了当前项目目标（即本周项目开发目标）
- 对run.py进行了函数化整理。把逻辑拆成了函数，符合开发规范
- 原来程序是按顺序直接往下执行，现在拆成了更清晰的几个函数：
  - `load_eval_questions()`：读取题集
  - `count_categories()`：统计题目类别
  - `validate_eval_questions()`：检查题集字段
  - `build_result(topic)`：构造结构化输出
  - `main()`：组织主流程
- 对 `README.md` 进行了补充和重写，使其更像正式项目首页
- 在 README 中加入了项目简介、当前状态、路线图、目录结构、运行方式、输出协议和下一步计划
- 将 README 里的绝对路径链接改成了相对链接
- 在 README 中补充了对学习路线图 HTML 文件的说明，明确它与 `docs`、`journal` 和代码文件之间的关系

### 过程中遇到的问题
- 一开始不太确定 `project_definition.md` 应该按天写，还是按周写
- 对 README 的定位有过摇摆：它既需要满足固定要求，又需要快速说明整个项目
- `run.py` 虽然已经能工作，但还不够像“可继续扩展的 baseline 程序”

### 我当时是怎么想的
- `project_definition.md` 更适合作为按周维护的稳定文件，而不是每天都跟着 journal 变化
- Day 2 的重点不是继续加功能，而是把 Day 1 形成的最小程序整理清楚
- README 不只是给人看目录，更应该帮助未来的自己和 AI 快速理解项目

### 关键决策
- 决定将 `project_definition.md` 作为按周更新的阶段性定义文件
- 决定把 `run.py` 函数化，而不是继续在单文件脚本里追加逻辑
- 决定将 README 调整为更像正式开源项目首页的结构
- 决定在 README 中明确说明学习路线图 HTML 的定位：它是项目的上层参考，而不是普通附件

### 留下的证据
- 命令：
- 文件：run.py readme.md project_definition.md
- 输出：
- `project_definition.md` 已形成按周视角的项目定义
- `run.py` 已改成函数化 baseline 程序
- `README.md` 已能说明项目状态、目录结构与周路线
