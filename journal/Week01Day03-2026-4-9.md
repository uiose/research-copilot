## 二、过程记录
### 我做了什么
- 为 Day 3 的最小 LLM 接入准备了本地 Python 环境与依赖安装方式。
- 最终决定统一使用 `uv` 作为当前项目的虚拟环境和依赖安装工具，避免后续再切换工具链。
- 在项目根目录执行了 `uv venv`，成功创建（或重建）了 `.venv` 虚拟环境，并确认后续可通过 `.\.venv\Scripts\Activate.ps1` 激活。
- 为 `run.py` 接入了最小可用的 LLM 调用路径，使用环境变量读取 `GITHUB_TOKEN` 和 `MODEL_NAME`。
- 把原本手写的结果构造逻辑替换成“提示词 -> 模型输出 -> JSON 解析”的链路，同时保留 `topic / assumptions / next_step` 这套固定输出协议。
- 保留了 Day 1 和 Day 2 已经搭好的 baseline 能力，没有破坏题集读取、题目分类统计和字段检查这条链路。
- 新增了 `.env.example` 和 `requirements.txt`，让 Day 3 的运行前提更清楚，避免模型调用部分只停留在代码里。
- 对模型输出增加了解析和字段校验，避免模型多说一句话时整个 baseline 直接失效。
- 修正模型调用地址后，成功跑通了一次真实的 GitHub Models 在线调用。
- 实际输入测试主题为 `s`，程序返回了符合协议的 JSON，包含 `topic`、`assumptions`、`next_step` 三个字段。

### 过程中遇到的问题
- system prompt编写不是很好，不理想。
- 模型调用本身比手写 `build_result()` 更脆弱，最明显的问题是模型可能输出额外解释、代码块，或者字段不完整。
- 最初模型调用失败，报 `400 Bad Request`，后面确认是 API 地址不对，修正后才成功调用。
- 虽然已经成功调用，但使用 `s` 这种过于模糊的输入时，返回内容只能说明结构是通的，内容质量还不具代表性。

### 我当时是怎么想的
- 既然这个项目后面会越来越工程化，那就尽早统一工具链，减少后续在 `venv / pip / uv` 之间切换的成本。
- Day 3 的目标不是一下子把 Agent 做复杂，而是先证明“真实模型调用”可以在现有 baseline 里安全落地。
- 所以我选择不去引入检索、工具调用和 LangGraph，而是先把最小闭环收紧成：读题集、输入主题、调用模型、校验结构、打印 JSON。
- prompt 即使现在还不够理想，也先以“守住固定输出协议”为首要目标，因为这会直接影响后续评测和批量运行。

### 关键决策
- 统一使用 `uv` 作为当前项目的虚拟环境和依赖安装工具。
- 继续沿用 `topic / assumptions / next_step` 作为 Day 3 的唯一输出协议，不在这一天扩字段。
- 通过 `.env` 注入凭据，而不是把 token 或模型名写死在代码里。
- 把“模型原始文本输出”和“最终可接受 JSON”分开处理，用解析函数单独守住结构边界。
- 把 Day 3 定义为“已接入 LLM 的 baseline”，而不是“已经具备检索能力的研究型 Agent”。
- 在出现 `400 Bad Request` 后，优先检查和修正调用端点，而不是贸然修改主流程其它部分。

### 没完成什么

- 还没有使用更有代表性的研究主题来留下一份更高质量的样例输出。

### 今天学到的东西
- 接入 LLM 不是把 API 调通就结束，真正关键的是调用后怎样把输出重新收回到固定协议里。
- baseline 阶段越早明确“允许输出什么、不允许输出什么”，后面的评测和迭代成本越低。
- `.env`、依赖和异常处理这些运行细节，本质上也是 Day 3 交付的一部分，不只是附属工作。
- `uv venv` 用于创建项目虚拟环境，`Activate.ps1` 用于激活环境，`uv pip install -r requirements.txt` 用于安装依赖。

- GitHub Models 上最终应固定哪一个模型作为 Week 1 的默认模型。
