主线项目
Research Copilot（面向研究生的文献调研 Agent）
这不是“课程目录”，而是一条围绕同一个系统持续演进的研究路线： 从 最小可运行 baseline 出发，逐步加入检索、记忆、规划、反思、评测、安全、可观测性， 最终目标不是“做出一个看起来很聪明的 demo”，而是形成 可写研究备忘录与论文提纲 的 Agent 原型。
默认主栈（尽量不换）
证据化回答 长期记忆 显式规划 反思与评测 安全与 HITL 可观测性与消融
北极星指标
减压执行原则
  • 每周只攻一个核心问题：先让系统可运行，再让它可评测，最后再追求更复杂的能力。
  • 技术栈尽量不换：12 周内默认用 Python + LangGraph + Chroma，减少无效工程切换。
  • 阅读坚持“三层法”：必读 1 个、选读 1–2 个、扫读若干摘要，避免资料过载。
  • 优先留下可见证据：日志、表格、失败案例、trace 截图，比“我感觉更强了”更有价值。
论文导向里程碑
  • Week 1–2：写清楚任务定义、输入输出、baseline 与固定题集，解决“系统到底在做什么”这个最重要的问题。
  • Week 3–4：形成 paper library、标签体系和 related-work 草表，为后续综述与实验打底。
  • Week 5–6：写出方法草图：planner / memory / retrieval / synthesis 如何协同。
  • Week 7–8：建立 benchmark、rubric 与误差分析表，形成论文里的“实验设置”和“错误分析”骨架。
  • Week 9–10：补安全与 human-in-the-loop 章节，把系统限制与风险写清楚。
  • Week 11–12：完成 trace、消融、研究备忘录和 4–6 页论文提纲，把 demo 整理成研究原型。

本周工作台

这里的内容会自动保存到本地；导出当前 HTML 时，也会把这些记录一起打包进去。

完成情况 / 当前系统状态

建议写：本周到底做成了什么、还差什么、系统现在是什么形态。

遇到的问题 / 卡点

建议写真实阻塞：环境、数据脏、prompt 不稳、工具 schema 不统一、评测难定义等。

实验记录

用最少的格式把实验留住：名称、对比对象、控制变量、结果、结论。

Prompt 优化记录

不要只保存“最后有效的 prompt”，还要保存“为什么这样改、改完效果怎样”。

失败案例记录

这是后续误差分析、导师汇报和写论文最值钱的材料之一。

论文 / 相关工作笔记

建议写：本周读到的关键论文、最像 baseline 的工作、可借鉴的方法、和你系统的差别。

下一步 / 下周计划

只写 1–3 条最关键动作，避免把待办写成“另一个大项目”。
建议每周结束至少留下：一张对比表、一个失败案例、一次 prompt 改动记录、一句下一步假设。这些内容到第 12 周会直接变成你的研究备忘录素材。
全局研究备忘录

把 12 周工作收束成论文与汇报材料

这里不是日记，而是持续收敛“研究问题—方法—实验—局限—下一步”的区域。建议从第 3 周开始就写，不要等到最后一天。

当前研究问题

一句话写清楚：你到底在研究什么，不要写成“我想做一个很厉害的 Agent”。

当前核心假设

写成可验证假设，而不是愿望。

Related Work 地图

记录：最相关的 3–5 篇工作、你的系统和它们最大的差异是什么。

论文提纲

建议从第 8 周开始逐步补：Introduction / Method / Experiments / Error Analysis / Limitations / Future Work。

给导师 / 组会的更新摘要

控制在 5–8 句话，方便每周/双周汇报。

最终摘要草稿

到第 12 周时，尝试写一个 150–250 字摘要。
